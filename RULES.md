# RatingsDQ 自定义上传版 · 规则与校验设计

> 目标：从「后台固定一个 Excel + 按账号过滤」改为「每个使用者上传自己的 Excel，
> 自适应识别列 → 校验 → 自动计算到期提醒」。可对接微信小程序 + Python 后端。

---

## 一、架构总览

```
微信小程序 ──上传 xlsx(bytes)──▶ Python 后端
                                  │
                       rating_engine.process_upload()
                       ├─ 零依赖读 xlsx（zipfile+XML，仅首个工作表）
                       ├─ 自适应列识别（承揽立项 / 项目作业 / 通用模糊）
                       ├─ 校验（必填列 / 日期可解析 / 空行）
                       └─ 返回 映射建议 + 校验报告 + 清洗记录
                                  │
                       merge_and_compute()
                       ├─ 多文件合并（按 openid；此处演示按文件）
                       ├─ 去重（受评主体 + 评级类型，取最新基准日）
                       ├─ 到期 / 提醒 / 三态计算
                       └─ 过期窗口过滤（去陈年噪音）
                                  │
                          返回 到期提醒结果（overdue/due/upcoming）
```

身份隔离：小程序用微信 `openid` 作隔离键，每人数据集独立；管理员 openid 可跨用户聚合总览。

## 二、两种已验证的导出格式

样本实测（同一业务人员刘鹏/浙江分公司的两个导出模块，**62/65 受评方重叠**）：

### 格式 A · 承揽立项（立项申请全集）
| 标准字段 | 实际列 | 备注 |
|---|---|---|
| 受评方 | `受评人` (J) | 空 11.6%，用 `委托方`(F) 兜底 |
| 基准日 | `申请时间` (X) | 可解析 241/259 |
| 状态 | `审核状态` (Z) | **有效=「已通过」**(224)，其余排除 |
| 评级类型 | `业务类型` (L) | **空 56.8%，不可靠**，仅展示 |
| 业务人员 | `申请人` (H) | 刘鹏为主 |

### 格式 B · 项目作业（派单/结项子集）
| 标准字段 | 实际列 | 备注 |
|---|---|---|
| 受评方 | `受评主体` (E) | 空 0.6%，`委托方`(F) 兜底 |
| 基准日 | `派单日期` (B) | 可解析 146/157，范围 2021~2026 |
| 状态 | `作业进度` (C) | **有效=「结项」/「结项,结项」**(112)，其余排除 |
| 评级类型 | `子项目债券类型` (H) | **0% 空，可靠**（主体/债项/公司债…）|
| 业务人员 | `营销人员` (K) | 刘鹏 155 |

> 关键判断：两张表是**同一人的互补数据**。《项目作业》的 `派单日期` 比《承揽立项》
> 的 `申请时间` 更接近真实评级时点，且带"结项"状态和可靠的债券类型。
> 因此默认**支持双文件合并**，合并后结果最准。

## 三、列识别规则（自适应 + 模糊兜底）

引擎按 `标准字段 → 候选表头（按优先级）` 匹配，归一化（去空格/标点/小写）后
先做精确匹配、再做包含匹配，输出置信度 high/medium。

- 受评方候选：`受评人 / 受评主体 / 被评级方 / 评级对象 / 客户名称 / 主体名称`
  - 兜底：`委托方 / 委托人`
- 基准日候选（优先级高→低）：`评审日期 / 评级日期 / 报告日期` → `派单日期 / 申请时间 / 立项时间`
  - 兜底：若都无，则 `年份 + 月份` 拼成当月 1 号
- 状态候选：`审核状态 / 作业进度 / 审批状态 / 状态`
- 评级类型候选：`债券类型 / 子项目债券类型 / 业务类型 / 项目类型`
- 业务人员候选：`申请人 / 营销人员 / 市场人员 / 业务员 / 负责人`

> 低置信度（medium）或缺列时，前端应提示用户**确认/手动改映射**，确认后存为该用户模板。

## 四、校验规则（上传即校验）

1. **可解析**：仅读第一个工作表（`.xlsx`）；非 xlsx 直接报错。
2. **必填列**：必须识别到「受评方」和「至少一个日期列」，否则 error 终止。
3. **丢弃行**：受评方为空（且委托方也空）**或** 基准日无法解析 → 丢弃并计数。
4. **日期可解析率**：低于阈值（如 <80%）warning，提示检查格式。
5. **状态分布**：展示各状态计数，明确哪些纳入、哪些排除。
6. **输出**：`映射建议 + 校验报告 + 清洗后记录`，可直接回前端做预览。

---

## 五、去重规则（核心）

- **去重键** = `受评主体 + 评级类型`（主体/债项是不同评级，各自算到期）。
  - 评级类型缺失（如承揽立项的业务类型空）→ 退化为「仅受评主体」。
- **每组取基准日最新一条**；基准日相同则优先数据源优先级高者
  （项目作业 zuoye=2 > 承揽立项 chenlan=1）。
- 这与原 RatingsDQ「同一客户取最新评审日」逻辑一致，并扩展到"主体+债项"。

## 六、到期与状态规则

- **到期日** = 基准日 `+ validity` 个月（默认 12，可配置）。
- **提醒起始** = 到期日 `- remind` 个月（默认 3，可配置）。
- **三态**：
  - `overdue`：参考日 > 到期日（已过期）
  - `due`：参考日 ≥ 提醒起始（3 个月内到期，优先跟进）
  - `upcoming`：其余（有效期内）
- **参考日**：默认 `date.today()`，后端可传参（演示用 2026-07-09）。

### 过期窗口（噪音过滤，重要）
历史评级可能回溯到 2021 年，全量 overdue 是噪音。引入 `overdue_window`
（默认 12 个月）：仅保留「过期未超 N 个月」的近期过期记录用于**重新营销**，
丢弃陈年过期项。最终视图 = `due(17) + upcoming(51) + 近期overdue(24)` = 92 条。

## 七、待你拍板的产品决策（默认值已给，可改）

| 决策点 | 当前默认 | 说明 |
|---|---|---|
| 是否合并双文件 | **合并**（推荐） | 单文件也能跑；合并最准 |
| 状态过滤 | 立项=已通过 / 作业=结项 | 其他状态不进提醒 |
| 去重键 | 受评主体 + 评级类型 | 类型缺失退化为仅主体 |
| 有效期 | 12 个月 | 是否按评级类型不同（如跟踪评级更短）？|
| 提醒窗口 | 3 个月 | 提前多久提醒 |
| 过期窗口 | 12 个月 | 过期多久内还提示重新营销 |
| 到期基准精度 | 派单/申请日 | **最准应是「评级报告日」**，两表均无 → 用派单日近似；若系统另有报告日列需补 |

> ⚠️ 最高优先级待确认：**真实评级报告出具日**是否存在于某个导出列？
> 若有，到期基准用它（比派单/申请日准 1~2 个月）。当前两表无此列。

## 九、管理员后台源数据（项目查询导出.xls）+ 归属模型（新增，已验证）

用户提供的第三份文件 `项目查询导出.xls`（46,492 行 × 31 列）是**全量评级台账**，
解决了上面的「出具时间」问题，并引出「客户跨年易主如何归属」问题。

### 9.1 关键列（实测）
- `[0] 主体名称` — 规范客户（对标「评级报告客户」），1.4% 空，7,472 唯一主体。
- **出具时间候选**：`报告落款日`(col20, 66.3% 有值) > `评审日期`(col17, 43.8%) > `打印报告日期`(col28, 49.1%)。
  → **以「报告落款日」为准，缺失回退 评审日期 → 打印报告日期**。
- `[13] 立项日期` — 100% 有值，用于时间窗口归属匹配。
- `[9] 负责人 / [11] 主笔` — **机构内部分析师（331/209 唯一值），不是外部市场人员**。
- 同一主体 4,561 个有多条记录（跨年初评/跟踪/重出），后台自身就存在跨年换负责人。

### 9.2 架构重定义（混合模型）
- **后台源数据 = 评级「真相」**：subject + 出具时间 + 项目类型/债项类型 → 算全部到期。
- **前端用户上传 = 「归属/所有权」真相**：受评主体 + 市场人员 + 申请/派单基准日。
- 二者按 (主体名称 + 时间窗口) 关联，把每条后台评级**归属**给正确市场人员。
- 每个用户只看到「归属给自己」的评级提醒；管理员看全部（总览）。
- 自服务兜底：若管理员尚未上传，用户仍可用自己文件算粗略到期（见前八节 engine）。

### 9.3 归属算法（两级，已原型验证 → admin_source.py）

**主机制：合同号 join（确定性，最精准）**
- 三份文件都含「合同编号」且同格式（`GC-B-0000-B-xx-xxxxx`）：
  合同管理.xlsx[A]、项目作业导出.xlsx[V]、后台项目查询导出.xls[27]。
- 前端上传 合同管理.xlsx 自带 `申请人`(市场人员) + `委托方` + `债券类型` + `合同状态`，
  后台源数据带 `合同编号` + `主体名称` + `出具时间`。
- **按合同编号精确 join** → 每条后台评级确定性归属到对应市场人员，
  同时校验 委托方≈主体名称、债券类型一致。无需模糊匹配客户名。
- 已验证：合同管理 132 有效合同号 ∩ 后台 = 103；join 示例
  `GC-B-0000-B-26-10627` → 申请人=刘鹏/委托方=杭州西湖区科创，后台主体同名 ✅。
- 实测归属：后台 45,846 条 → 经合同号成功归属 141 条（刘鹏140/黄天祺1），其余为无合同号旧记录。

**兜底机制：客户名 + 时间窗口（处理无合同号记录）**
```
对每条无合同号的后台评级记录 R(主体名称 S, 立项日期 D, 出具时间 T):
  1. 取前端用户上传中 subject==S 的候选市场人员列表 (openid, 基准日)
  2. 选基准日落在 [T - 窗口(默认24月), T] 内、且最接近 T 的市场人员
     —— 即「发起该评级项目的市场人员」
  3. 无人命中 → unassigned（仅管理员可见）
```
**为什么对**：业务上某年级的评级由当年立项的市场人员驱动；
窗口匹配天然实现「一年内一般一个市场人员」，并正确拆分「23年我/25年别人」。
（原型验证：演示客户「四川省乡村发展集团」2022→A_刘鹏、2024+→B_他人 ✅）

**结论：合同号 join 为主、时间窗口兜底**，二者结合覆盖全部记录。

### 9.4 原型验证（真实数据）
演示客户「四川省乡村发展集团」：
```
立项=2022-12 出具=2022-12-30 → 归属 A_刘鹏   (23年我做)
立项=2024-06 出具=2024-08-29 → 归属 B_他人   (25年别人做)
立项=2025-03 出具=2025-07-21 → 归属 B_他人
...
```
同客户跨年易主被正确拆分到不同市场人员 ✅

### 9.5 前端上传格式建议（重要）
- **首选 `合同管理.xlsx` 作为市场人员前端上传**：它同时含
  `合同编号`(唯一键) + `申请人`(市场人员) + `委托方` + `债券类型`(可靠,91主体等)，
  一个文件即可完成"精准 join + 所有权 + 债项类型"，无需模糊匹配。
- `项目作业导出.xlsx` 也含 `合同编号`[V]，可作备选上传。
- `承揽立项 (1).xlsx` **无合同编号、业务类型空56.8%**，最弱，仅作无合同号时的兜底。
- 解析时需过滤：占位合同号（如"客服中心分配合同编号"，26个）、`已作废/已终止`状态。

### 9.6 待确认/注意
- **有效期是否按项目类型不同**：初评/更新=12月；定期跟踪可能更频。默认12月，可配置。
- **无合同号记录兜底**：用 9.3 时间窗口；主体名称需归一化（去空格、统一（）/()、去尾部年份）+ 可选模糊匹配 + 用户确认。
- **unassigned**：后台有评级但合同号不在任何上传中 → 仅管理员可见，待补传。
- `.xls`(后台) 用 xlrd 解析（后端场景）；`.xlsx`(前端上传) 用零依赖引擎；前端引擎 rating_engine.py 仍零依赖。
- 约 14,247 条后台记录无出具时间（流程中/未出报告）→ 不进到期提醒，正确。

## 十、后端服务骨架（新增，已端到端验证）

把上面规则封装成可独立部署的 HTTP 服务，支撑「小程序 + 市场人员自传 + 管理员总览」。
完整启动/API/微信接入/部署说明见 **`README_server.md`**，关键点：

**归属两级机制（已验证）**：
- **主 · 合同号 join**：后台评级记录的 `合同编号` 直接 join 前端上传的 `合同管理.xlsx`，
  精确归属到对应 openid —— 解决「同客户跨年易主」难题（不同年份=不同合同号，各归各上传者）。
- **兜底 · 客户名+时间窗口**：无合同号、或合同号不在上传范围的记录，用 (受评主体, 基准日)
  匹配前端上传的承揽/作业，取落在「出具时间−24月 ~ 出具时间」内、最接近的 marketer。
- **未匹配 (unassigned)**：仅管理员可见，待补传。

**实测（参考日 2026-07-09）**：后台 39603 条 → 合同号 join 命中 134 条（全 `contract_join`）；
加兜底后补充若干 `window_match`/`closest_fallback`；剩余 `no_candidate` 仅管理员可见。
A(刘鹏)/B(其他) 按 openid 严格隔离，`/api/my/ratings` 与 `/api/admin/overview` 均通过。

**新增文件**：
- `db.py` — SQLite 存储层（users / admin_source / contract_uploads / fallback_uploads / final_ratings）
- `server.py` — Flask API（login / 上传 / compute / 个人查询 / 管理员总览）
- `test_smoke.py` — 端到端冒烟测试（真实数据，全部断言通过）
- `requirements.txt` — Flask / flask-cors / xlrd
- `README_server.md` — 启动与对接文档
